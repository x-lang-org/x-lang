## 架构设计说明

### 两阶段后端模型

```
┌─────────────────────────────────────────────────┐
│         LIR（统一低级中间表示）                 │
│     所有十大后端的统一输入来源                   │
└─────────────────────────────────────────────────┘
              │
              ├─────────────────────┬──────────────────
              │                     │
        【第一阶段】          【第二阶段】
      翻译为既有格式         直接生成格式
              │                     │
    ┌─────────┼─────────┐        ┌──┴─────────┐
    │         │         │        │            │
 源码翻译   IR翻译   字节码    优化生成
  (7个)     (1个)    (1个)     (各后端)
    │         │         │        │
TS/Java   LLVM      Erlang  Java/C#/
Python/           BEAM    Erlang/
Zig/C#/Swift/Rust           Native
```

### 十大后端分布

| 类型 | 后端 | 数量 | 优先级 | 现状 |
|------|------|------|--------|------|
| 源码翻译型 | TS、Java、Python、Zig、C#、Swift、Rust | 7 | 🔴 P0/P1 | 🚧 早期/✅ Zig成熟 |
| 中间IR翻译型 | LLVM | 1 | 🔴 P0 | 🚧 早期 |
| 字节码翻译型 | Erlang | 1 | 🟠 P2 | 📋 规划 |
| 机器码直接生成型 | Native | 1 | 🟠 P2 | 📋 规划 |

## 关键设计原则

### 1. 统一 LIR 输入
所有十大后端都从同一个 LIR 开始，确保一致性

### 2. 两阶段演进
第一阶段翻译获得可用性，第二阶段直接生成提升性能

### 3. 模块化组织
每个后端独立文件或模块，易于维护和扩展

### 4. 充分利用现有工具
调用 tsc、javac、rustc 等成熟编译器，减少重复实现
