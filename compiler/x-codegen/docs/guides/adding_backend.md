## 概览

本指南指导你如何为 X 编译器添加一个新的后端。

### 后端的职责

一个后端需要完成以下任务：

1. **接收 LIR 模块** - 获取来自编译流程的低级中间表示
2. **转换为目标格式** - 将 LIR 翻译为目标语言源码、中间 IR 或字节码
3. **输出结果** - 保存转换后的内容到文件或返回给调用者
4. **集成工具链**（可选）- 调用目标工具链（编译器、解释器等）完成最终产物

### 架构两个阶段

**第一阶段（当前）**：翻译为既有工具链支持的格式
- 充分利用 tsc、javac、rustc 等成熟编译器
- 快速原型开发，验证概念可行性

**第二阶段（未来）**：直接生成目标格式
- 某些后端演进为直接生成字节码或机器码
- 跳过中间步骤，提升编译速度和输出质量

impl XxxBackend {
    fn map_type(&self, x_type: &str) -> Result<String, String> {
        match x_type {
            "i32" => Ok("int".to_string()),
            "i64" => Ok("long".to_string()),
            "f32" => Ok("float".to_string()),
            "f64" => Ok("double".to_string()),
            "bool" => Ok("boolean".to_string()),

#### 3.3 实现全局变量生成

```rust
impl XxxBackend {
    fn codegen_globals(&self, output: &mut String, module: &Module) -> Result<(), String> {
        if module.globals.is_empty() {
            return Ok(());
        }
        
        writeln!(output, "// Global variables").map_err(|_| "write error".to_string())?;
        for global in &module.globals {
            let ty = self.map_type(&global.ty)?;
            writeln!(output, "let {}: {} = ...;", global.name, ty)
                .map_err(|_| "write error".to_string())?;
        }
        writeln!(output).map_err(|_| "write error".to_string())?;
        
        Ok(())
    }
}
```

#### 3.4 实现函数生成

```rust
impl XxxBackend {
    fn codegen_functions(&self, output: &mut String, module: &Module) -> Result<(), String> {
        for func in &module.functions {
            self.codegen_function(output, func)?;
        }
        Ok(())
    }
    
    fn codegen_function(&self, output: &mut String, func: &Function) -> Result<(), String> {
        // 生成函数签名
        let params = func.params.iter()
            .map(|p| format!("{}: {}", p.name, self.map_type(&p.ty).unwrap_or_default()))
            .collect::<Vec<_>>()
            .join(", ");
        
        let return_type = self.map_type(&func.return_type)?;
        
        writeln!(output, "fn {}({}) -> {} {{", func.name, params, return_type)
            .map_err(|_| "write error".to_string())?;
        
        // 生成函数体
        for block in &func.blocks {
            self.codegen_block(output, block)?;
        }
        
        writeln!(output, "}}").map_err(|_| "write error".to_string())?;
        writeln!(output).map_err(|_| "write error".to_string())?;
        
        Ok(())
    }
}
```

### 步骤 4：集成工具链

如果你的后端生成源代码，需要调用相应的编译器：

```rust
impl XxxBackend {
    /// 调用目标编译器编译生成的源码
    pub fn compile(source_file: &str, output_file: &str) -> Result<(), String> {
        use std::process::Command;
        
        let output = Command::new("xxx-compiler")
            .arg(source_file)
            .arg("-o")
            .arg(output_file)
            .output()
            .map_err(|e| format!("Failed to run compiler: {}", e))?;
        
        if !output.status.success() {
            let stderr = String::from_utf8_lossy(&output.stderr);
            return Err(format!("Compilation failed: {}", stderr));
        }
        
        Ok(())
    }
}
```

## 常见问题

### Q1: 从哪里开始最简单？

**A**: 选择 TypeScript 或 Python。原因：
- 类型系统相对简单
- 工具链成熟稳定
- 文档齐全，容易上手

### Q2: 如何调试代码生成问题？

**A**: 使用以下方法：
1. 打印生成的源代码：`eprintln!("{}", generated_code);`
2. 手工验证生成的源代码是否有效
3. 使用目标工具链的 lint 工具（如 `tsc --noEmit`）
4. 使用调试器逐步跟踪代码生成过程

### Q3: 如何处理 X 语言的特性（如效果系统）？

**A**: 映射到目标语言的等价机制：
- X 的 `Result<T>` → Java 的异常或 `Optional`
- X 的效果系统 → 目标语言的特性标注或签名
- X 的 Perceus → 目标语言的内存管理模型

### Q4: 支持 Wasm 需要做什么特殊处理？

**A**: 
- 了解 WebAssembly 的限制（如没有垃圾回收）
- 特殊处理指针和内存管理
- 考虑使用专门的 Wasm 后端（如通过 Zig）

### Q5: 第二阶段（直接字节码生成）如何规划？

**A**: 当第一阶段稳定后：
1. 学习目标格式（如 JVM 字节码格式）
2. 直接生成字节码而不是源码
3. 跳过目标编译器调用
4. 性能通常会显著提升
